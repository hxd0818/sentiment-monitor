# 搜索迭代方法论 v3.1

> **核心理念：没有积累就没有迭代，没有迭代就没有深挖。**
> **本文件定义的不是"怎么思考"，而是"怎么让思考可积累、可传递、可进化"。**

---

## 一、核心机制：状态文件是迭代的物理基础

### 1.1 为什么必须有状态文件

Agent的上下文窗口会被压缩。压缩后，前面轮次的分析细节会丢失。
**状态文件 = Agent的外部记忆 = 跨轮次的思考连续性保障。**

不是"可选草稿纸"。是**必须维护的迭代中枢**。

### 1.2 状态文件的职责

每轮循环结束时，Agent **必须** 更新状态文件，记录：

| 记录什么 | 为什么需要 |
|---------|-----------|
| 这轮搜了什么 | 避免重复搜索 |
| 发现了什么新实体/事件/信号 | 种子池增长 = 迭代的证据 |
| 对每个发现的判断 | 下轮不需要重新分析 |
| 基于这些发现，下一步想搜什么方向 | 下轮的起点 |
| 收敛判断的依据 | 后续可以回溯为什么停 |

**如果一轮搜索结束后没有更新状态文件，这一轮等于白做。**

## 二、状态文件设计

### 2.1 文件：`status.json`

位置：`data/{品牌名}/status.json`

```json
{
  "brand": "品牌名",
  "version": "v3.1",
  
  "round_info": {
    "current_round": 0,
    "last_action": "初始化/搜索/分析/深挖/收敛/完成",
    "last_updated": "ISO时间"
  },

  "seed_pool": {
    "_meta": "种子池是迭代的引擎。每轮基于种子池决策，每轮结束更新种子池。",
    "seeds": [
      {
        "id": "S-001",
        "type": "event|person|company|keyword|platform|fingerprint",
        "value": "种子值",
        "source_round": 1,
        "status": "active|matured|dormant|exhausted",
        "depth": 0,
        "children": ["S-003", "S-004"],
        "findings": "这个种子相关的关键发现摘要",
        "next_directions": ["基于此种子还想探索的方向1", "方向2"]
      }
    ]
  },

  "risk_events": [
    {
      "id": "RE-001",
      "title": "事件标题",
      "level": "L1|L2|L3|L4",
      "status": "tracking|deep_diving|verified|resolved",
      "related_seeds": ["S-001", "S-002"],

      "unknowns": ["还不知道什么(=下一轮的搜索方向)"],
      "timeline": [{"time": "", "event": "", "source": ""}],

    }
  ],

  "search_log": [
    {

      "intent": "为什么搜这些（本轮搜索目标）",
      "results_summary": "发现了什么（一句话）",
      "urls_found": [                                    // ★ 新增：本轮发现的所有有价值URL
        {"url": "https://...", "title": "标题", "relevance": "high|medium|low", "tagged_to": "RE-001/S-003/通用"}
      ],
      "new_seeds": ["S-001", "S-002"],
      "new_risks": ["RE-001"],

    "_meta": "所有有价值的URL的中央索引。写报告时直接从这里取，不用翻raw文件。",
    "total_count": 0,
    "by_relevance": {
      "high": [],   // 高价值URL（直接点名品牌的负面/央媒/权威信源）
      "medium": [], // 中等价值（行业媒体/间接相关）
      "low": []     // 低值（UGC/转载/仅提及）
    },
    "by_event": {   // 按风险事件索引
      "RE-001": [],
      "RE-002": []
    },
    "by_sentiment": {
      "positive": [],
      "neutral": [],
      "negative": []
    }
  },

  "convergence": {
    "is_converged": false,
    "reason": "",
    "evidence": {
      "rounds_with_low_gain": 0,
      "last_significant_finding_round": 0,
      "unexplored_directions": []
    }
  },

  "hidden_state": {
    /* 隐匿发现子体系的完整迭代状态。
       含: lens_log(每次透镜执行记录) / coverage_map(7透镜覆盖状态)
       / evolution_chain(链式演化记录) / convergence(隐匿子体系收敛判断)
       完整schema见 references/hidden-risk-discovery.md §二 */
  }
}
```

### 2.2 关键字段的设计意图

**seed_pool.seeds[].next_directions** — 这是**扩散的核心物理载体**。
不是抽象的"种子会分裂"，而是具体地记录："围绕S-001这个事件，我还想查金额细节、涉及哪些县、监管方态度"。

下轮开始时，读这个字段就知道上轮想到了什么还没做。

**risk_events[].unknowns** — 这是**深挖的核心物理载体**。
明确记录"还不知道什么"。每轮的目标就是消灭unknowns。

**search_log[].information_gain + next_thinking** — 这是**收敛判断的数据基础**。
连续多轮low/none gain + next_thinking越来越空洞 → 该收敛了。

## 三、迭代循环（带状态机）

### 3.1 真正的迭代长什么样

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│  ① 读 status.json（恢复记忆）                        │
│     "上次我想查X方向，种子池里有A/B/C，D还在追踪中"     │
│                                                     │
│  ② 基于状态 + 新思考，决定本轮搜索方向                │
│     "D事件的unknowns里还有'金额'没查清，优先查这个"     │
│                                                     │
│  ③ 执行搜索                                         │
│                                                     │
│  ④ 读结果，分析                                     │
│     "找到了！金额是498万，还发现涉及海城市"            │
│                                                     │
│  ⑤ 更新 status.json（写入本轮所有发现）              │
│     - D事件新增子种子: 金额498万, 海城市             │
│     - unknowns删除"金额",新增"合同细节"               │
│     - search_log追加本轮记录                          │
│     - information_gain: high                         │
│     - next_thinking: "查合同细节+海城市背景"          │
│                                                     │
│  ⑥ 判断：继续？还是收敛？                            │
│     gain=high + 有明确next → 继续下一轮              │
│     gain=low + next模糊 → 考虑收敛                  │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 3.2 扩散是如何发生的

**扩散不是主动的"向四周发散搜索"，而是被动的"一个发现引出更多问题"。**

具体机制：

```
第N轮发现: "某咨询公司因百强县榜单问题被通报"
  │
  ├── 这个发现自动产生疑问（unknowns）:
  │   "哪家咨询公司？" → 搜品牌名确认 ✓
  │   "多少钱？" → 新方向！→ 下轮搜金额
  │   "哪些县？" → 新方向！→ 下轮搜地理
  │   "谁通报的？" → 新方向！→ 下轮搜监管方
  │   "还有类似情况吗？" → 新方向！→ 下轮搜扩展
  │
  └── 这些疑问写入 risk_event.unknowns 和 seed.next_directions
      → 下轮读状态时自然看到
      → 基于这些方向设计Query
      → 这就是"扩散"
```

**种子分裂的物理实现**：

```
S-001 (event): "百强县买榜事件" [active, depth=0]
  │ 第2轮深挖后:
  ├─ S-003 (keyword): "498万咨询费" [active, depth=1, parent=S-001]
  ├─ S-004 (location): "海城市" [active, depth=1, parent=S-001]
  ├─ S-005 (person): "涉事领导" [active, depth=1, parent=S-001]
  └─ S-006 (company): "多个合作县市" [active, depth=1, parent=S-001]

  S-003 第3轮深挖后:
  └─ S-007 (keyword): "咨询合同细节" [active, depth=2, parent=S-003]

depth字段记录挖掘深度。depth越大说明挖得越深。
parent字段记录来源，可以追溯"我为什么要搜这个"。
```

### 3.3 收敛是如何判断的

**收敛不靠轮数，靠信息增益趋势 + 方向枯竭度。**

每次更新status.json时填写 `convergence` 段：

```json
{
  "convergence": {
    "is_converged": false,
    "evidence": {
      "rounds_with_low_gain": 2,       // 连续2轮gain=low/none
      "last_significant_finding_round": 3, // 最后一次重要发现在第3轮
      "unexplored_directions": []       // 所有next_directions都已探索
    },
    "reason": ""  // 收敛时填写原因
  }
}
```

**收敛条件（同时满足才收敛）**：
1. 连续2-3轮 information_gain 为 low 或 none
2. seed_pool 中所有 active 种子的 next_directions 要么已探索要么价值很低
3. risk_events 中所有 tracking 状态的事件 unknowns 要么已解答要么无法在线获取

**只要有一个活跃种子还有高价值的未探索方向，就不收敛。**

## 四、Query设计哲学

### 4.1 Query从哪里来

**好的Query不是凭空设计的，是从状态文件中的未知数生成的。**

```
❌ 凭空设计: "我应该搜'赛迪顾问 负面'" （泛泛而谈）
✅ 从未知数生成: risk_event.unknowns=["具体金额"] 
   → Query: "百强县 买榜 金额" / "咨询费 多少钱"
```

### 4.2 三种Query来源

| 来源 | 什么时候用 | 示例 |
|------|-----------|------|
| **消灭unknowns** | 每轮主力 | 事件unknowns驱动 |
| **激活dormant种子** | 主力方向枯竭时 | 之前降级但可能复活的种子 |
| **广度嗅探** | 每轮开头快速扫 | 品牌名本身看最新动态 |

### 4.3 设计原则

- **脱品牌名**：最危险的隐匿文章不用品牌名。用事件指纹搜
- **全文验证**：snippet不可信。高相关结果必须抓全文
- **小步快跑**：每轮3-5个精准Query > 1轮10个大而泛的Query
- **跟随线索**：上轮结果中的新线索 = 下轮最佳输入

## 五、风险发现的思考框架

### 5.1 风险层级

| 层级 | 定义 | 发现方式 |
|------|------|---------|
| 显性风险 | 直接点名品牌的负面 | 常规搜索即可 |
| 隐匿风险 | 不点名品牌但可推断关联 | R1-R7思考透镜 |
| 潜在风险 | 尚未发生但有迹象 | R7反事实推演 |
| 级联风险 | A事件触发B事件的连锁反应 | 关联方扫描 |

### 5.2 隐匿性发现的7个思考透镜

不要当成检查清单。当成**审视盲区的不同角度**：

| 透镜 | 核心问题 |
|------|---------|
| **R1 语义伪装** | 如果有人想说不点名品牌的负面，他会用什么词？ |
| **R2 实体辐射** | 谁跟这个品牌有关系？这些人有没有出事？ |
| **R3 时间共振** | 在这个事件前后，有没有异常的信息聚集？ |
| **R4 跨域融合** | 除了舆论，其他领域有没有同步异常？ |
| **R5 渠道下沉** | 主流媒体之外，有没有人在讨论？ |
| **R6 框架转移** | 同一件事，别人可能用什么更危险的框架叙述？ |
| **R7 反事实推演** | 如果情况恶化，下一步会怎样？有先兆吗？ |

**使用方式**：当你觉得"好像没什么负面了"的时候，逐个透镜过一遍——不是每个都搜一轮，而是问自己"从这个角度看，有没有遗漏？"有就深挖，没有就跳过。

### 5.3 框架升级检测

当一个事件被发现后，它的叙事框架可能会随时间升级。

常见路径（参考）：
```
行业讨论 → 商业争议 → 行政违规 → 形式主义典型 → 系统性乱象 → 腐败/犯罪案例 → 教育素材
```

**检测方法**：用更危险的框架词 + 事件特征组合搜索，看是否有新框架下的文章出现。

## 六、时效性原则（⚠️ 核心约束）

### 6.1 目标定位

**本系统的目标是预警风险、发现最新负面信息。不是写企业史。**

| 信息类型 | 处理方式 | 原因 |
|---------|---------|------|
| **近3个月内的负面** | 深挖+扩散+隐匿扫描全上 | 核心目标 |
| **3-12个月的事件** | 关注是否有新进展/框架升级 | 可能复燃 |
| **1年以上的旧事件** | 仅背景提及，不扩散深挖 | 资源浪费 |
| **3年以上的历史事件** | 除非被最新文章重新激活，否则忽略 | 与预警无关 |

### 6.2 时效标记

种子和风险事件必须标注时效：

| time_tag | 定义 | 是否触发扩散/隐匿扫描 |
|----------|------|---------------------|
| `current` | 3个月内 | ✅ 全力 |
| `recent` | 3-12个月 | ✅ 适度 |
| `stale` | 1-3年 | ❌ 不扩散，仅跟踪新进展 |
| `historical` | 3年以上 | ❌ 忽略（除非被reactivated） |

### 6.3 决策规则

- **优先级**：current > recent > stale > historical
- **扩散限制**：只有 current/recent 级别的种子才触发子种子分裂和 unknowns
- **隐匿聚焦**：R1-R7 主要围绕近期事件
- **历史事件**：报告背景段一笔带过，不分配搜索资源
- **重新激活**：若2026年新文章引用旧事件作为"惯犯"证据 → 升级为 reactivated

### 6.4 收敛中的时效信号

> 如果连续2轮搜索的主要结果都是1年以上旧信息 → **强烈应该收敛了**。

```json
"convergence": {
  "evidence": {
    "stale_ratio": 0.6,  // 最近一轮陈旧信息占比，>50%该收敛
    "current_risks_fully_mined": true/false
  }
}
```

## 七、脱品牌名追踪（⚠️ 必选步骤，非可选）

### 7.0 为什么必须有这一步

**这是整个系统最关键的防漏机制。没有它，所有基于品牌名的搜索都是盲区。**

> **核心事实**：当一个事件被中央/国家级信源通报后，它不再只是"品牌的危机"，而是"公共议题"。公共议题的二次/三次传播几乎不用品牌原名。

**真实案例（2026-04-21 漏报）**：
- 文章：《新型腐败正在体制内悄悄蔓延：不收钱不收礼，但却更隐蔽更加值得警惕》
- 该文将赛迪顾问百强县买榜案列为 **"6种新型腐败之第4种——虚假合作型"**
- **全文从未出现"赛迪顾问"4个字**，用"某百强县榜单评价机构"代称
- Query `赛迪顾问 新型腐败` → ❌ 搜不到
- Query `新型腐败 榜单 百强县` → ✅ 命中

### 7.1 触发条件（满足任一即触发）

| 触发信号 | 说明 |
|---------|------|
| 事件被 **中央/国家级** 信源通报 | 中纪委、国务院、人民日报、新华社等 |
| 事件被 **省级以上** 纪委/监委点名 | 升级为廉政/反腐叙事 |
| 事件的叙事框架出现 **"形式主义→牟利→腐败"** 路径升级 | 框架在持续恶化 |
| L2级及以上风险事件 | 隐匿风险高 |

### 7.2 执行方法

**步骤1：提取事件指纹（与品牌完全无关的独立识别词）**

```
原始事件："赛迪顾问因百强县买榜被中纪委通报"
     ↓ 提取指纹
事件指纹 = ["百强县", "榜单", "498万", "买榜", "海城", 
           "县域经济", "形式主义", "咨询费"]
```

**规则**：
- 指纹词必须是不包含品牌名、股票代码、任何直接标识的词
- 优先提取：金额、地名、行业术语、行为动词、官方文件用语
- 指纹写入 status.json 的 seed_pool（type="fingerprint"）

**步骤2：设计脱品牌名Query**

```
模式A - 指纹 + 叙事框架组合（最常用）:
  "新型腐败 榜单 百强县"
  "虚假合作 形式主义 县域经济"
  "买榜 牟利 咨询费"

模式B - 指纹 + 时间/来源组合:
  "2025 百强县 通报 形式主义"
  "498万 咨询 榜单"

模式C - 纯框架搜索（最激进）:
  "新型腐败 6种 类型"
  "隐形腐败 不收钱"
```

**关键规则**：
- **Query中绝对不能出现品牌名、简称、股票代码**
- 用 sentiment-collect.py 执行时 --round 标记为 `debrand-N`
- 结果中的文章需要做**实体归因**：这篇文章说的"某机构"/"某公司"是不是我们的目标？

**步骤3：实体归因判断**

对每条脱品牌名搜索结果，回答：
1. 这篇文章是否在描述我们的目标事件？（对比指纹细节）
2. 如果是，它用了什么代称/隐含指代？
3. 它把事件放在什么叙事框架下？（可能比我们已知的更危险）
4. **如果确认关联 → 立即抓全文（--fetch-url），提取完整分析**

### 7.3 在迭代循环中的位置

```
① 读 status.json
② 决定本轮方向
③ 执行常规搜索（带品牌名）        ← 正常流程
④ 分析结果
⑤ 【必选】检查是否需要脱品牌名追踪 ← ★ 新增强制步骤
   ├─ 有L2+事件/中央通报？→ 是 → 执行§七脱品牌名追踪
   └─ 没有？→ 跳过
⑥ 更新 status.json（含指纹种子和脱品牌名搜索记录）
⑦ 收敛判断
```

**执行时机**：
- **首次触发**：当发现事件符合§7.1触发条件的那一轮**立即执行**
- **后续跟踪**：每轮迭代时检查是否有新的框架升级信号，有则再次执行
- **最低要求**：每次报告生成前，至少执行**1轮**完整的脱品牌名追踪

### 7.4 status.json 中的记录

```json
{
  "seed_pool": {
    "seeds": [
      {
        "id": "S-F001",
        "type": "fingerprint",
```

## 八、URL中央注册机制

### 8.1 为什么需要

**报告中的URL缺失是质量硬伤。** 原因：URL散落在raw/文件中，写报告时Agent需要逐个翻找——容易遗漏、容易偷懒。

**解法：每轮搜索后立即把有价值的URL提取到 status.json 的 url_registry 中。写报告时直接从 registry 取用。**

### 7.2 操作规则

**每轮搜索结束后，执行以下动作（与状态更新同步完成）：**

1. **读本轮 raw 文件**，提取所有结果中的 URL
2. **对每个URL判断**：
   - relevance: high（直接相关负面/央媒/权威）| medium（行业媒体/间接）| low（转载/UGC）
   - tagged_to: 关联到哪个风险事件/种子？（RE-001 / S-003 / "通用"）
   - sentiment: positive / neutral / negative
3. **写入两处**：
   - `search_log[round].urls_found` — 本轮发现的URL列表
   - `url_registry` — 按relevance/event/sentiment三个维度索引

### 7.3 url_registry 格式

```json
{
  "url_registry": {
    "total_count": 15,
    "by_relevance": {
      "high": [
        {"url": "https://...", "title": "...", "round": 1, "tagged_to": "RE-001", "sentiment": "negative"}
      ],
      "medium": [...],
      "low": [...]
    },
    "by_event": {
      "RE-001": ["url1", "url2"],
      "RE-002": ["url3"]
    },
    "by_sentiment": {
      "positive": ["url4"],
      "neutral": ["url5"],
      "negative": ["url1", "url2", "url3"]
    }
  }
}
```

### 7.4 写报告时的使用方式

```
写第二章（详细分析）→ 从 url_registry.by_event["RE-001"] 取URL
写第六章（信息源表）→ 从 url_registry.by_relevance["high"] 开始填
写6.4隐匿信源    → 从 url_registry 中筛选 hidden_risks_found 关联的URL
```

**原则：registry中有的URL必须出现在报告中。registry中没有的URL不允许凭空编造。**
