# 报告写作规范与自检清单 v2.1

> **严格对齐 report-template.md 的8章结构。**
> **核心：格式正确 + 无禁用词 + 信源完整。**

---

## 8章速查（顺序不可变）

| # | 章节名 | 必须包含的内容 |
|---|--------|---------------|
| 0 | **报告概要** | 生成时间 / 监测周期 / 数据来源（只列媒体和网站名称，如"新华网、人民网、新浪财经"） |
| 一 | **核心发现** | 1.1 整体舆情评估（必须包含：正面X% / 中性X% / 负面X%，三者加起来=100%）+ 1.2 核心指标表（信息源总数/高关注度数/高影响力数/正负比例） |
| 二 | **详细分析** | 2.1正面 / 2.2中性 / 2.3负面。每节格式：(a)主题概括 (b)代表性信息列表，每条格式为[标题](URL) (c)分析段落 |
| 三 | **话题维度分析** | 3.1产品服务 / 3.2管理治理 / 3.3财务业绩 / 3.4社会责任 / 3.5争议危机。每节格式：舆情表现评级 + 主要观点概述 |
| 四 | **风险评估** | 4.1 高风险关注点（每个风险含：标题/严重程度/影响范围/应对建议列表）+ 4.2 潜在风险预警（同上格式） |
| 五 | **应对建议** | 5.1短期(1-7天) / 5.2中期(1-4周) / 5.3长期(1-3月)。每条格式：建议标题 + 目标 + 具体执行方案 |
| 六 | **信息源清单** | 6.1正面信息源表 + 6.2中性信息源表 + 6.3负面信息源表 + 6.4隐匿性风险信源表。四张表都必须存在 |
| 附录 | **附录** | A.数据来源说明(只列渠道不列工具) / B.分析方法 / C.免责声明。末尾固定写：报告生成日期 + 技术支持: 赛迪网 |

## 禁止事项与正确做法对照

| # | 禁止做什么 | 应该怎么做 |
|---|-----------|-----------|
| 1 | 写"百度AI搜索""web_fetch""Brave搜索""通过XX搜索发现"等 | 直接写信息内容，来源只写媒体名。例如："据新华网报道[URL]"而不是"通过百度AI搜索找到新华网报道" |
| 2 | 写"Sentinel Monitor""v1.3""Agent驱动舆情分析""赛迪网舆情监测系统v2"等技术支持文字 | 固定只写四个字：**赛迪网**。放在报告末尾"技术支持:"后面 |
| 3 | 增加第9个章节，如"执行摘要""情感总览""PIR矩阵""PIR履行情况""主题热力图""情绪驱动因子""影响力图谱""渠道热度映射""竞品对比"等 | 只写上面列出的8章（含附录）。需要额外分析内容时，嵌入到对应章节中，不要新建章节 |
| 4 | 正文第二章引用了某条具体信息，但第六章对应的表格中找不到该信息 | 每当在2.1/2.2/2.3的"代表性信息"列表中加入一条时，同时在6.1/6.2/6.3的表格中增加一行。两条必须对应 |
| 5 | 第六章只有3张表（缺少6.4隐匿信源表） | 即使没有发现隐匿信号也要写6.4节标题和表头，无数据则写"本期未发现隐匿性风险信号" |
| 6 | 信息源的URL栏为空或编造URL | 找不到真实URL就写"URL缺失: [原因]"。绝对不要编造一个看起来像真的URL |
| 7 | 建议写成空话如"加强管理""提高意识""高度重视" | 每条建议必须包含具体的行动步骤。例如："在7日内通过官方渠道发布事件调查声明"而不是"加强危机公关" |

## 自检清单（写完报告后、转PDF前逐条检查）

```
[ ] 章节检查: 用grep "^## "扫描报告，结果恰好为以下9行（含附录）:
     ## 报告概要
     ## 一、核心发现
     ## 二、详细分析
     ## 三、话题维度分析
     ## 四、风险评估
     ## 五、应对建议
     ## 六、信息源清单
     ## 附录
    多一行或少一行都不合格。

[ ] 禁用词检查: 运行 grep -i "百度AI\|web_fetch\|Brave搜索\|Sentinel Monitor\|Agent驱动\|baidu-search" report.md
    结果必须为0（没有任何匹配行）。如果有匹配，修改后再检。

[ ] 技术支持检查: 运行 grep "技术支持" report.md
    结果所在行必须包含且仅包含"赛迪网"四个字。

[ ] 情感分布检查: 第一章1.1节必须包含三个百分比数字（正面/中性/负面），三者相加必须等于100%。

[ ] URL完整性检查: 第六章四张表的每一行都必须有URL列的值。允许"URL缺失: xxx"但不允许留空。

[ ] 数据沉淀检查: data/{品牌}/raw/ 目录下必须有>=6个搜索结果文件。
                   data/{品牌}/status.json 必须存在且 round >= 6。
                   data/{品牌}/hidden_risks.json **必须存在**（隐匿性发现必产出）。

[ ] 隐匿扫描完成度检查: hidden_risks.json 中必须包含R1-R7每个策略的扫描记录。
                           即使某策略无发现也要注明"该策略未发现隐匿信号"。
                           如果文件写"经七策略全扫描未发现额外隐匿信号"，确认七个策略都有提及。

[ ] 建议可执行性检查: 第五章每条建议都能回答"谁、在什么时间前、做什么、做到什么程度"。
```

全部勾选通过后才执行 PDF 转换命令。
